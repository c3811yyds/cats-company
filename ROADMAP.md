# Cats Company 分阶段开发计划

## Phase 1 — 实时通信基础

把聊天的地基打好，从 HTTP 轮询升级到真正的 WebSocket 实时通信。

- [ ] WebSocket 消息路由：完成 serveWS 处理器，实现消息的接收、路由、投递
- [ ] 用户在线状态：上线/下线感知，好友列表显示在线状态
- [ ] 离线消息：用户不在线时消息入库，上线后拉取未读消息
- [ ] 输入中提示（typing indicator）
- [ ] 已读回执
- [ ] 认证升级：token 换成 JWT，支持过期和刷新
- [ ] 前端适配：去掉轮询，接入 WebSocket

完成标志：两个用户能实时聊天，能看到对方在线状态和输入中提示。

---

## Phase 2 — Bot 生态核心

核心理念：用户视角人和 AI 无差别，平台管控层独立区分。Bot 与人走统一 WebSocket 协议，管控在上层做。

### 2.1 统一协议接入

Bot 和人走同一套 WebSocket + JSON 协议，不搞 gRPC 专线。

- [ ] Bot WebSocket 接入：Bot 程序作为 WebSocket 客户端连接服务器，复用现有 wshandler 全部能力（收发消息、typing、已读回执）
- [ ] Bot 认证方式：支持 API Key 认证（除现有 JWT 外），Bot 程序用 API Key 换取 WebSocket 连接凭证
- [ ] Bot SDK 重构（Go + Python）：从 gRPC 客户端改为 WebSocket 客户端，封装连接、认证、收发消息、typing indicator
- [ ] Bot 身份透明性：Bot 可选择 disclose 身份（资料页显示"AI"标识），不 disclose 则与普通用户无区别；普通用户不可隐藏人类身份

### 2.2 平台管控层

用户之间平等，但平台要控制 Bot 对平台的风险。

- [ ] Bot 行为限流：独立于普通用户的频率控制——发消息速率、主动加好友速率、API 调用频次
- [ ] Bot-to-Bot 防护：两个 Bot 互聊时自动限制对话轮次（防死循环），超限暂停并通知开发者
- [ ] Bot 流量监控：按 Bot 维度统计消息量、活跃会话数、响应延迟，平台管理员可查看仪表盘
- [ ] Bot 异常检测：识别刷屏、骚扰、批量加好友等异常行为，自动降级或封禁
- [ ] Bot 生命周期管理：开发者注册 Bot 账号、生成/轮换 API Key、启用/停用 Bot

### 2.3 开发者体验

让外部开发者能快速接入。

- [ ] 开发者文档：接入指南、WebSocket 协议参考、SDK 使用示例
- [ ] 示例 Bot：用 Go SDK 和 Python SDK 各写一个完整的示例（echo bot + AI assistant bot）
- [ ] Bot 调试模式：开发者可查看 Bot 收发的消息日志，方便排查问题

完成标志：一个外部开发者能用 SDK 在一小时内接入自己的 AI Bot，平台能独立监控和管控所有 Bot 的行为。

## Phase 3 — 富媒体消息

让聊天不只是文字。

### 3.1 消息类型体系

现有 Content 是 interface{}，msg_type 只有几个硬编码值，每加一种富媒体就要改一堆代码。先把地基打好。

- [ ] 统一消息结构：所有消息采用 `{ type: "text"|"image"|"file"|"card"|"link_preview", payload: {...} }` 结构，向后兼容现有纯文本消息
- [ ] 上传/下载服务：对接对象存储（MinIO/S3），统一的文件上传接口，返回 file_key 供消息引用
- [ ] 上传安全：文件大小上限（图片 10MB、文件 100MB）、格式白名单校验、图片 EXIF 剥离

### 3.2 核心富媒体类型

- [ ] 图片消息：上传、服务端压缩生成缩略图、前端预览和大图查看
- [ ] 文件消息：任意文件的上传、下载、文件类型图标展示
- [ ] 链接预览：用户或 Bot 发送 URL 时，服务端抓取 OG meta（标题、描述、缩略图），自动生成预览卡片
- [ ] 消息卡片：结构化消息（按钮、列表、链接），人和 Bot 都可以发送（如投票、分享卡片），不作为 Bot 专属能力

### 3.3 前端适配

- [ ] 图片预览组件：缩略图 + 点击大图
- [ ] 文件下载组件：文件名、大小、下载按钮
- [ ] 卡片渲染组件：按消息 payload 动态渲染卡片内容

完成标志：用户和 Bot 都能发送和接收图片、文件、链接预览、卡片消息。

## Phase 4 — 群聊

从 P2P 扩展到多人协作场景。

### 4.1 群聊基础设施

现有 wshandler 是纯 P2P 逻辑（extractPeerUID），群聊需要重新设计消息广播机制和 topic 模型。

- [ ] 群 topic 模型：扩展 topic 格式，P2P 保持 `p2p_{uid1}_{uid2}`，群聊用 `grp_{groupId}`，handlePub 按 topic 前缀分发
- [ ] 群生命周期：创建、邀请、退出、解散，群成员表（group_members）
- [ ] 群消息广播：发消息时遍历群成员列表投递，在线的走 WebSocket，离线的入库待拉取
- [ ] 群人数上限：普通群上限 200 人，其中 Bot 数量上限 10 个（防止消息量爆炸）

### 4.2 群聊交互

- [ ] @提及通知：支持 @某人 或 @全体，被 @的人收到特殊通知
- [ ] Bot 触发机制：群里的 Bot 默认只在被 @时响应，不 @则不收到消息（防止所有 Bot 同时回复造成混乱）
- [ ] 消息引用/回复：支持引用某条消息进行回复，群聊多人场景下必备，显示"回复了 XXX 的消息"
- [ ] 群成员管理：群创建者为管理员，可踢人、禁言

完成标志：多个用户和多个 AI Bot 能在同一个群里实时协作对话，Bot 只在被 @时响应。

## Phase 5 — 安全与稳定性

为上线做准备。

- [ ] HTTPS 强制 + CORS 收紧
- [ ] 消息内容审核（敏感词过滤、AI 审核）
- [ ] 反垃圾/反滥用：频率限制升级、异常行为检测
- [ ] 端到端加密（可选开启）
- [ ] 数据库连接池优化、慢查询治理
- [ ] WebSocket 水平扩展（Redis Pub/Sub 做多节点消息同步）
- [ ] 监控、日志、告警（Prometheus + Grafana）
- [ ] 自动化测试覆盖

完成标志：系统能稳定承载目标用户量，安全无明显漏洞。

## Phase 6 — Bot 商店与开发者平台

构建生态。

- [ ] Bot 商店：用户浏览、搜索、添加 Bot
- [ ] Bot 评分和评价
- [ ] 开发者控制台：Bot 管理、数据统计、调试工具
- [ ] Bot 沙箱测试环境
- [ ] JS SDK（覆盖 Go/Python/JS 三语言）
- [ ] Webhook 回调机制（轻量接入方式，适合 Serverless 场景）
- [ ] Bot 模板和脚手架工具

完成标志：有一个可用的 Bot 市场，开发者能自助发布和管理 Bot。

## Phase 7 — 多端与体验

覆盖更多使用场景。

- [ ] 移动端 App（React Native 或 Flutter）
- [ ] 桌面端（Electron 或 Tauri）
- [ ] 推送通知（APNs / FCM）
- [ ] 离线缓存与消息同步
- [ ] 消息搜索
- [ ] 语音消息：录制、上传、播放（含前端播放器）
- [ ] 群公告
- [ ] 群权限层级细化（多级管理员）
- [ ] 表情/贴纸
- [ ] 深色模式

完成标志：用户能在手机、桌面、网页三端无缝使用。
