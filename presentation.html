怎么走到这一步的

起点

我们做了 XiaoBa，挂在飞书上跑，体验不错。但飞书限制太多 —— 消息格式受限、API 能力有限、没法自由分发和售卖。我们需要一个自己控制的平台。

平台

所以我们做了 Cats Company —— 一个 AI 友好的社交平台。Agent 以聊天窗口的形态存在，用户加好友就能用。不是 API marketplace 的一次性调用，而是持续的对话关系。

记忆

对话窗口下的 Agent 要能无限聊下去。服务重启 session 丢了，上一条消息都不知道；聊几天后远超 context window，更不可能回忆。要像人一样聊天，就必须有记忆。

闭环

我们恰好有 GauzMem。记忆赛道看似鸡肋 —— mem0 这类服务实际上没人真正在用，多数只是无代码平台挂个插件。但对话平台上需要无限沟通的 Agent bot，是记忆最好的 playground。三件事自然串成了闭环。

Cats Company 社交平台

为什么是社交平台，而不是 API marketplace

统一协议

Bot 和人使用同一套 WebSocket 协议，同一套消息格式。在系统层面，Agent 就是一个用户，不是挂载的插件。

第三方接入

Bot SDK（Go + Python），API Key 认证。第三方开发者接入自己的 Agent，不需要了解平台内部实现。

分发 = 加好友 = 变现

用户浏览 Agent、加好友、在聊天窗口直接使用和付费。Agent 的分发和售卖变成社交行为，不是 API 调用。这是我们当前最直接的变现路径。

API marketplace 是一次性调用关系。社交平台提供的是持续对话关系 —— 这才是 Agent 真正需要的分发形态，也是最自然的付费场景。

GauzMem 记忆引擎

记忆赛道的真实卡位

现状：Claude Code、Cursor 这类 agent 都摒弃了记忆管理，靠 tool-based search + context window + 模型自身能力来处理文档。mem0 等记忆服务实际上没有真实用户 —— 多数只是无代码平台挂个插件，没有产生实际价值。

但在对话平台上，Agent 需要在一个聊天窗口里无限沟通下去。服务重启 session 就没了，聊几天就超出 context window。这个场景下，记忆不是锦上添花，是刚需。Cats Company 是 GauzMem 最好的 playground。

核心设计：写入快，消化慢，回忆自然

Fast Path · 同步秒级

信息进来，同步完成四件事就返回：存原文、LLM 提取原子事实 + 显式关系、生成 embedding、标记待代谢。
MySQL + Neo4j + Qdrant 三层同步写入

Metabolism · 异步消化

ReAct 模式的 LLM Agent 按"不舒适度"游走：消歧（"他"→"John"）、建联想关系、合并重复、标记低质量。
不是无限堆积，是有选择地记住

Retrieval · 按需回忆

被动回忆：语义搜索 → 图扩展 → 连通分量聚类，渲染成树形结构填入 prompt。
主动搜索：作为 tool 给 agent 调用，返回 raw JSON。

为什么是三位一体

不是设计出来的组合，是需求自然推导出来的闭环

→

平台带来真实场景：用户在聊天窗口和 Agent 持续沟通，产生了对记忆的刚需。没有平台，记忆服务就是 mem0 —— 没有真实用户。

→

记忆让 Agent 可用：有了记忆，Agent 才能在对话窗口里像人一样持续沟通。用户留存和体验直接取决于记忆质量。

→

Runtime 降低供给门槛：XiaoBa 让开发者（包括我们自己）快速部署 Agent。更多 Agent 入驻，平台生态更丰富。

→

对第三方同样成立：任何第三方 Agent 接入平台后，都可以挂载 GauzMem 获得记忆能力。记忆服务从我们自用变成平台基础设施，可以收费。
